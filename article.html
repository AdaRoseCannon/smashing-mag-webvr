Virtual Reality for Web Developers

Abstract: Recently, there is a proliferation of VR Web Browsers and VR capabilities added to traditional browsers. This article looks at the state of browsers in VR and the state of VR on the Web via the WebVR APIs. Finally, we have a look at how a web developer with can feel comfortable working in the new medium via the popular library A-Frame, which uses the custom elements API to allow HTML to markup 3D scenes which can be controlled via familiar JavaScript.

State of Virtual Reality and the Web

The web has experimented with VR before with VRML, but now WebVR takes a new approach to VR, one more suited to the modern web. We've had accelerated 3D in the web since 2011 with the release of WebGL. Now the Web can handle VR thanks to new web APIs for taking advantage of virtual reality hardware using WebGL.

These APIs enable WebGL content to be displayed in 3D with a VR headset. They also provide headset and controller tracking information to give the user presence in the virtual world.

WebVR was first developed in 2014 at Mozilla. In 2016, an earlier version of the standard was available for Desktop Chrome, Firefox and Samsung’s virtual reality web browser: Samsung Internet for Gear VR.

These days, the standard is incredibly well supported on phones and desktop computers for almost all major headsets.

The WebVR standards are worked on in the open and represent a collaboration between Mozilla, Google, Samsung, Oculus, Microsoft and, recently, Apple.

This means that a single website that uses WebVR can make an immersive scene and deliver it to all major VR platforms at once, Desktop and Mobile!

The web's ability to handle virtual reality content allows one to easily share a VR experience in a URL and view it in the browser without the complicated app stores and long download times associated with native VR.

Additional VR-related Web APIs in Samsung Internet

These APIs are not part of the WebVR API but are useful for viewing immersive content in traditional websites in an immersive web browser without needing to use WebGL.

These APIs are being developed for Samsung Internet for Gear VR. We hope they will be picked up by other browsers and standardised.

Panoramic Video

This is the ability to play panoramic videos (monoscopic and stereoscopic) immersively by setting the type="dimension=360;" attribute on a video tag. These videos also get enhanced in Samsung Internet 5.0 by allowing the user to pan around within the video using their fingertips.

Possible values:

  • dimension=3d-lr: Side-by-Side 3D Video
  • dimension=3d-tb: Top-to-bottom 3D Video
  • dimension=360: 360˚ Video
  • dimension=360-lr: Side-by-side 3D 360˚ Video
  • dimension=360-tb: Top-to-bottom 3D 360˚ Video
  • dimension=180: 180˚ Video
  • dimension=180-lr: Side-by-side 3D 180˚ Video
  • dimension=180-tb: Top-to-bottom 3D 180˚ Video
<video src=”/360.webm" type=”video/webm; dimension=360;”></video>

Changing the Sky

Another API available in Samsung Internet for Gear VRis a JavaScript API for changing the background image of the VR web browser to one of the developer’s choosing.

Your traditional 2D website is still visible but the surroundings are set to one which matches the environment of your site.

window.SamsungChangeSky({ sphere: 'http://site.com/blue-sky.jpg' });

What is WebVR?

WebVR is a set of cross-browser JavaScript APIs which provide a variety of VR related utilities to place the user into an immersive environment generated using WebGL.

By providing a side-by-side rendered 3D images, these APIs will handle all of the complexity involved in displaying an undistorted stereoscopic 3D image to the user.

I won’t go into details of implementing the standard here as the standard is still changing. Most users will never need to deal with them directly, as often WebGL tools and libraries include handling the WebVR APIs for you.

State of the WebVR APIs

Current version of the APIs is known internally as version 1.1. Version 2.0 will change some of the method names and remove some unused methods.

The precise details of the APIs can be read on MDN and the standards are discussed on Github. There is interest from within the WebVR community group and the W3C to move to a W3C working group once they are ready.

Essentially what the WebVR API provides is:

  • Headset Tracking to allow the user to look around in the Virtual Environment. This comes with frame interpolation built in so that it is always tracking the user’s head even if you skip the occasional frame.
  • 6 degrees of freedom and 3 degrees of freedom controller support allowing controllers like those of the HTC Vive and the Gear VR to work in virtual reality. Allowing the user to use their hands to interact with the virtual environment.
  • Information about how the headset needs the 3D information rendered, such as field-of-view and how to position the render for each eye on the canvas.
  • A new headset specific requestAnimationFrame synced to the refresh rate of the display in the headset.
  • A method of submitting the rendered frames to the headset in the form of a WebGL enabled canvas element.

What does it mean to build a WebVR Immersive Experience?

Surprisingly, building a virtual reality website raises many of the same problems as building a good mobile website or progressive web app.

Starting Fast

One of the biggest problems being tackled on the web today is network performance, which is important because:

  • User’s attention spans are shrinking
  • Networks are getting more congested
  • Websites are getting bigger than ever before

WebGL and WebVR websites are certainly no exception to size; they can be very large if one is not careful!

Right now VR content has an edge over traditional content because it is novel and interesting enough that users will likely wait a bit longer to have a go. Still it is extremely important to get your 3D experience started in under a few seconds. Users are impatient and only getting more so.

Before your VR capable site has loaded it is just a 2D site and a promise of cool things to come.

My advice here is to not preload everything, load just enough for the user to get started, then dynamically load and start caching the rest. This behaviour should be familiar with those who have read about Google’s PRPL pattern.

Even just showing a blurred 360 skybox and some low poly content allowing the user to look around will buy you precious seconds of engagement to bring in additional content and to bootstrap an engaging experience.

It is far better to show something basic but fast than lose the user because they got bored waiting for a loading bar to complete.

But bear in mind, network operations can be CPU intensive and block the main thread. This could give the user a bad experience if it happens a lot so it is a fine line to walk.

Perhaps one or two very intensive assets need to be preloaded to avoid breaking the experience, but if you have so many it is taking a long time to start perhaps it's worth thinking about finding a more performant alternative.

Making good use of a service worker and the Cache API to cache static assets for fast return visits is a great way to keep users coming back for more.

Progressive Enhancement

The two main platforms for VR are polar opposites: high-end desktop computers with advanced controllers; and mid-high level mobile phones, which may have only a single rotation tracked controller or no controller at all!

This presents us with two challenges:
  1. Maintaining a consistent frame rate across platforms with wildly differing performance capabilities.
  2. Presenting a user friendly experience across VR devices with a wide range of inputs.

Phones have by far the largest reach due to the popularity of the Gear VR and Daydream and the cheap price and high availability of the Google Cardboard headsets.

Below I have described some typical controller configurations; you don’t need to support them all but handling the 'no controller' situation as a baseline and also supporting another controller option if applicable will allow everyone to experience something. Supporting all controller configurations would be nice, but in my opinion not a reasonable expectation.

Some libraries such as Universal Controls from A-Frame Extras will try to make the best out of whatever is available.

Asymmetric Play

Like with progressive enhancement on the web, just because you support all levels of hardware it does not mean you need to deliver the same experience to all.

A user with two fully tracked hands in the virtual world will be able to have a much more engaging experience and should not be limited by being delivered the same experience as someone with no controller.

An example: A VR app where you produce VR artwork could use tracked controllers to produce art on high powered machines; on mobile a user would be able to view this art in VR, but be unable to edit it.

Another example: A networked VR multiplayer game could use tracked controllers to play the game; A mobile viewer could watch it and use gazed based interactions to choose different points of view.

Test as you develop

Like modern web design the key is to design mobile first. As you build your scene, regularly test it on real mid level smart phones with no controllers to ensure it will work for the majority of your users.

WebVR allows you to target both platforms at once but delivering the same content to both may result in mobile struggling or desktops not being used to their full potential.

There is no problem having very well performing graphics. A stylized low-poly appearance can look fantastic and render very quickly.

For upgrading graphics, one solution is to provide options for graphical quality before your user starts using WebVR. If the user requests high quality, then start downloading the large or difficult-to-render graphics.

Something more difficult, but more seamless, is to start on lowest graphical setting and detect how well the device is performing using commands like requestIdleCallback or measuring how long the render takes. If the device is running well then maybe increase the graphics settings. If frames start being skipped then dynamically reduce them.

When you are upgrading your scene you may decide to do a few things:

  • Increase draw distance.
  • Download and use high resolution models or textures
  • Use more complex shaders

This will ensure that users on mobile and desktop get the best possible experience. For most use cases though it is probably fine if the desktop users get the mobile experience as you can guarantee it will maintain great frame rate on the desktop if it also does on mobile.

After all there is more to a great scene than visual fidelity. Highly stylised games like Team Fortress 2 still look great today where as "realistic" games from the same era have not aged so well.

A great scene should have a well designed graphical style with bold colours and strong silhouettes. This will help low-power and low-resolution devices look good, but also still look great on desktop devices only needing a little bit of extra polish.

In VR most users will have the equivalent of poor eyesight so it is important to bare this in mind. This minimise uses of text or causes the user to strain their eyes to look at something.

What does the web bring to VR?

VR on the web looks to solve some of the issues VR currently has:

One of the biggest issues is that the user has to make a large commitment to use a one-off experience they may not want to come back to.

On native mobile and desktop VR one has to download the app from an app-store such as the Oculus store on Gear VR or from Steam for the HTC Vive or Oculus Rift.

This app-store pattern lends itself well to expensive video games where users have already invested some money so are going to return again and again. But for one-off experiences like shopping, viewing a movie, or trying a new social platform, this can be a large barrier to entry.

Users tend to be put off by the idea of having apps laying around on the device taking up space, or using up a large amount of data-allowance to download. Especially if the user is on a mobile device which may have limited disk space or network data caps.

On the web, once you leave the web page the user can stop worrying about content hanging around as the browser will clean it up if space is needed. Whilst there is space the developer can still cache content on the user’s device for when they return, letting you have your cake and eat it too.

Of course this requires the developer to make VR websites which are not stored as a single giant bundle, otherwise the gains the web can provide are lost.

By delivering VR assets dynamically and separately like those of a web page, you can take advantage of all the smart caching the web can provide from the CDN all the way down to the HTTP cache and the service worker's Cache API on the device.

Further more the user can jump straight into your VR experience with very little waiting.

A highly optimised WebVR website should render the first frame within a single second of the user landing on the website. Totally cutting out any lengthy initial downloads or app stores. Massively increasing engagement.

Experiences can be shared with nothing but a URL, which can be shared by social media or email, even written on a wall or a displayed on TV, making it much more likely your VR content will go viral due to it’s low barrier to entry.

Making the most out of the quick web

One feature WebVR sites often has is that before they enter VR you can usually view and interact with the scene on your 2D display. The view will often rotate with the phone giving the appearance of a magic window into a virtual space.

The magic window is a very powerful pattern. It allows users to get a preview of your VR scene without needing to put on their VR headset. This is great for when they don’t have their VR gear to hand or are on public transport and don’t want to wear VR equipment in public.

When the user has had a taste it will encourage them to bookmark your site to try in VR when it is more suitable for them.

The web has high level interfaces to low level APIs bringing advanced technology to any developer.

There are many web APIs you may have heard of or used which become massively more relevant in a virtual reality context.

WebSockets — Used to stream text and binary data to a server in real time. For VR they can be used to sync 100s of users in real time to enjoy a shared experienced and see Avatars representing each other. I recently produced an open source demo where I did this live for over 150 users at a conference.

WebRTC — To expand upon shared VR, one can also use WebRTC to maintain peer-to-peer connections for binary data and video/audio streaming. This can be used to allow two avatars to perform voice chat, or to sync the position and posture of a virtual reality Avatar without going through a central server. This can be used to connect 6–8 users at a time.

WebAudio — One of the most surprisingly powerful APIs. The browser contains everything you need for audio manipulation and analysis. One can even use a panner node for 3D spatial audio in VR. For producing virtual environments that give an immersive feel, WebAudio is more important than ever before.

SpeechRecognition — Newer browsers contain a built in speech recognition engine! Useful for giving commands and entering text when a real or virtual keyboard is difficult or unwieldy. There is a great example built for VR here: https://samsunginter.net/word-drop/

What effect might VR have on the web in the longer term?

Virtual Reality has already had an effect on the Web Platform, the WebVR APIs have been implemented across multiple platforms and there are discussions about creating a WebVR working group within the W3C.

Virtual Reality is already going mainstream and with augmented reality and mixed reality devices starting to enter the consumer realm, it is important the web is ready to take advantage of the new platforms.

WebVR as we know it today is entirely reliant on WebGL. Optimising for this WebGL use case will mean the browsers will have to look at taking advantage of the hardware’s optimisations to increase render speed towards that of a natively compiled app. Speed is important because dropping frames in VR can have disastrous effects, even to the point of making the user ill!

WebGL 2 will soon be reaching stable browsers. WebGL 2 brings WebGL closer to the OpenGL ES 3.0 spec. Greater visual fidelity and faster ways to do advanced graphics will make virtual reality truly an incredible visual experience.

WebAudio may be required to produce more accurate 3D audio transforms, known as head related transfer functions, to better reproduce the high quality 3D audio required by big budget productions. Accurate 3D Audio will be essential for delivering a high quality immersive video content such as 360 movies and immersive audio experiences.

Scripting on the web as well would benefit from significant performance improvements. There are a number of JavaScript APIs in the works which can be used to increase performance in the Web.

JavaScript itself can be optimised and precompiled. Another option is to compile other languages to WASM (Web Assembly). This can provide some speed ups across the board. Providing a bundle which is smaller to download and faster to parse and execute. If used wisely in a modular fashion, WASM could be used to make the core rendering engine of a WebVR experience, which we could still interact with using JavaScript like we do today.

The browser can make use of Web Workers to enable calculations which do not block the main thread. This is good because the main thread is primarily used for rendering. WebWorkers are useful for manipulating large amounts of data with CPU intensive calculations such as Physics Engines. By isolating the calculations from the main thread this way, they are less likely to trigger frame drops.

Unfortunately there is some cost associated with sending data to and from web workers to use them in the main thread this is alleviated partially by transferable objects. Transferable objects such as ArrayBuffers allow you to change the owner of the object but handling the transferring of this object can be difficult and error prone if a mistake is made.

I write about using Array Buffers in the Web here.

A new API called Shared Array Buffers will allow multiple workers to share the same Array Buffer useful for this use case.

On the topic of workers part of the issue is that right now the thread which renders the web page also needs to be used to render the WebGL scene. So any side effects of other code you run on the main thread such Garbage Collection or CPU bound functions may cause frames to be dropped.

OffscreenCanvas enables rendering to be performed in a web worker this will help the very important and sensitive render loop to be isolated from the other threads.

The other important rendering use case is prerecorded 2D and 3D video. These can be used as textures in WebGL. But they lack fine grain control like we have an <audio> element and an audioContext in JavaScript we will need to add a videoContext to enable performant video manipulation to assist with playing 360 videos in 3D.

An area where the Web and virtual reality currently lie at odds with the web as a whole is rendering documents in virtual reality. Displaying documents is the core functionality of the web platform but displaying a document in WebGL is nigh impossible without some very slow, very clever re-rendering.

It would be great to bring them together by having the browser expose rendered DOM content to WebGL, would allow us to take advantage of the Webs power for 2D interfaces but may potentially be a security/privacy risk!

Another path

WebGL based VR doesn’t necessarily have to be the future of VR on the web. Having to perform even the simplest use cases for WebVR in WebGL seems short sighted at best; and potentially fatal for VR on the web in the long run.

Part of the strength of the web and HTML lies in that it is a declarative language. Browsers can interpret it best according to the platform it is on. You don’t see the exact same website on your desktop computer as you do on your phone or TV. virtual reality is yet another platform in the variety of mediums to experience the web.

By being declarative, like HTML or CSS, VR on the Web could automatically handle rendering to balance render speed and visual fidelity. A high end computer could use advanced shaders and detailed models; a low power mobile phone could automatically use simple shading and pick low poly models. Much like the way the <picture> element can be used to download images which are the correct resolution and crop for the device it is on.

HTML could be extended to include some common virtual reality use cases, such as: Playing 360 degree/3D videos and images, displaying 3D models or moving bits of the webpage outside of the 2D viewport into 3D space.

Samsung has started to look at some of these use cases in the web browser, Samsung Internet for Gear VR.

It has inbuilt support for 3D video using the <video> element. Example display a Side by side 3D, 360 degree video is just a HTML tag:

<video controls src="360video.mp4" type="video/mp4; dimension=360-lr;">

The Middle Path

Of course these needn’t be mutually exclusive. The web can provide some ability to handle and optimise for simple VR use cases whilst also providing optimisations for building immersive VR from scratch with WebGL.

The Extensible Web manifesto is the idea that the web does not have to sacrifice extensibility for ease of use. That the community can use the low level tools provided to extend the web platform using libraries.

Virtual Reality is one case this vision seems very relevant. We already have the low level tools of WebGL and the WebVR APIs.

The library A-Frame provides custom HTML elements to build WebGL based 3D scenes. A-Frame is usable on it’s own or with popular frameworks such as React and Angular.

A-Frame allows any web developer with HTML experience to describe VR ready 3D scenes. To control them using familiar JavaScript. Even tools like JQuery, Angular and React can be used to change your scene because at the end of the day it is still just HTML.

Final Message

The web has the power to bring virtual reality to the world. To every consumer; to every developer.

It is still early days for VR in the web but now is the time to get building. To see what works and what does not.

The web can show that VR is for more than video games. Virtual reality can be used to enhance everything we currently do on the web and even enable new interactions only possible in an immersive medium.

As developers we can start building virtual reality experiences on the web today. By getting involved and giving feedback into the standards process we can ensure that VR on the web becomes a robust standard paving the way for future developers to build on.

Even if you don’t think VR is mature yet. With mixed reality and augmented reality devices round the corner what we build today will still be relevant then. The interface patterns we build for VR will apply to all immersive mediums. Don’t be left behind.

Together we will build the web of tomorrow.
